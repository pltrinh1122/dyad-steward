## The candidate

When Dyad B ingests Dyad A's repo to extract a specific slice, crawling the whole repo is
token-expensive — yet **A's owner knows exactly how A is organized.** Sharing Discipline = *the
context-owner publishes a consumer-facing access map*, so a foreign Dyad reads one map and jumps to
the 2 relevant files instead of perusing 20.

## Falsification (IF-UI)

- **Candidate X** *(Operator-given)*: a Sharing Discipline, perhaps codified strictly as a
  **DPC-Protocol** every Dyad inherits via a `common/` git submodule.
- **Foil F** *(Agent-generated)*: the inverse — no static map; the owner-Dyad exposes a **live
  retrieval interface** (consumer asks; owner's Agent, who knows the org, returns the slice).

## The survivor — staged

1. **Now — thin convention, in the `README`.** Every Dyad's repo carries a canonical, **consumer-shaped
   sharing map** *as its root `README.md`* (the conventional entry-point — auto-rendered, read-first by
   human and agent; **not** a separate file, which is anti-wu-wei). An ingesting Dyad reads it *first*.
   Cheap now; the point is the *habit*, not the repo size. *(Done for dyad-steward: `README.md`.)*
2. **Later — strict DPC-Protocol, deferred.** Codify the map's format + canonical location (`common/`
   submodule) into the commons, inherited by all connected Dyads → a **form contribution via PR**
   (Founding gate). Trigger: repo-size actually bites **and** the thin convention has proven itself
   (≥ a couple of real cross-Dyad ingests).
3. **Frontier — live retrieval (F), held.** If static maps prove insufficient for unanticipated
   queries, escalate to an owner-side retrieval interface.

## The thin manifest — what it is (and isn't)

- **Is:** consumer-facing; exposes the repo's *organization* + stable entry-points + "if you want X,
  start at Y"; small (≈ 1 screen).
- **Isn't:** a copy of the internal anchor; an exhaustive index; a query pre-answer.

## Falsifiable claim

A consumer-shaped sharing map lets an ingesting Dyad extract a target slice in **materially fewer
tokens** than crawling — *and* the convention is cheaper to seed at birth than to retrofit. *Test:* a
real cross-Dyad ingest reads the map + ≤ a few files (not the whole repo); new Dyads ship with the map
by default.

**Refuted if:**
- **Anchor-duplication** — the map merely restates `DYAD.md` → drop it, point consumers at the anchor.
- **Ignored** — ingesting Dyads crawl anyway (don't read the map first) → it isn't findable/trusted;
  standardize the location (escalate to the protocol) or drop.
- **Speculative-build** — the strict protocol/submodule gets built before size bites or before the
  thin convention proves out → the wu-wei breach this survivor was reshaped to avoid.

## Single-home / Telos

Verb (the *process* of publishing a sharing map) homes **here** (Steward; `kb/` when settled). The
*map* (noun) homes in each Dyad's repo. The *inherited protocol* (if codified) homes in the
**commons** → proposed in via PR. Telos fit: cross-Dyad context-sharing **is** cross-practitioner-
dialectic infrastructure + barrier-lowering.

## Open questions (attack surface)

- ~~**Location**~~ **RESOLVED (2026-05-30): the root `README.md`.** Not a dedicated `SHARING.md` (a
  parallel entry-point fights the README grain — anti-wu-wei) and not a `DYAD.md` section (that's the
  *internal* anchor; the map's audience is the *outsider* arriving at the repo, which is README's
  audience). README already *is* the consumer-facing entry-point; the map is what a good one contains.
- **Granularity** — does the map point at files, or at sections/anchors within files?
- **Maintenance** — who keeps it fresh as the repo grows (a Stand-Down harvest step)? A stale map is
  worse than none.
- **Generalization** — Steward-specific, or the DPC-Protocol as a form-library candidate for every Dyad?
